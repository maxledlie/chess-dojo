## Project Overview

"Cress" is a home-grown chess server. The repo has a Python/FastAPI backend and a React/TypeScript frontend.

## Commands

### Backend

Run from the `backend/` directory. Uses `uv` as the package manager (Python 3.14+).

```bash
# Start the backend (API + matchmaking daemon as separate processes)
cd backend && uv run python main.py

# Run all tests
cd backend && uv run pytest

# Run a single test
cd backend && uv run pytest tests/test_websocket_auth.py::test_successful_auth
```

The backend requires a `.env` file in `backend/` with:
- `SESSION_SECRET` — used to sign session cookies (itsdangerous)
- `REDIS_ENDPOINT` / `REDIS_PASSWORD` — Redis connection
- `GAME_REQUEST_TTL_SECONDS` — how long game requests live in Redis
- `ALLOWED_ORIGINS` — comma-separated WebSocket origins (default: `http://localhost:5173`)
- `ENV` — set to `prod` to enable `Secure` cookies

### Frontend

Run from the `frontend/` directory. Uses npm.

```bash
cd frontend && npm run dev        # Dev server (Vite, port 5173)
cd frontend && npm run build      # TypeScript check + Vite build
cd frontend && npm run lint       # ESLint
cd frontend && npm run generate-client  # Regenerate API client from OpenAPI schema
```

Vite proxies `/api` → `http://localhost:8000` and `/ws` → `ws://localhost:8000`.

## Architecture

### Backend Process Model

`backend/main.py` launches two Python `multiprocessing.Process` instances:

1. **API process** (`api/app.py`) — FastAPI on port 8000. Handles HTTP and WebSocket connections.
2. **Matchmaking daemon** (`matchmaking/daemon.py`) — reads from a Redis stream and pairs waiting players.

Both processes share state only through **Redis**, not shared memory.

### WebSocket Message Flow

All real-time communication uses a single WebSocket endpoint (`/ws`).

- **Auth**: Clients must first call `GET /session` to obtain a signed session cookie, which the WebSocket handshake validates via `guest_auth.py`.
- **Origin check**: The WS endpoint validates the `Origin` header against `ALLOWED_ORIGINS`.
- **Message envelope**: Every message is wrapped as `{ "data": { "msg_type": "...", ...fields } }`. The `msg_type` discriminates the Pydantic union type defined in `websocket/models.py`.
- **Per-connection queue**: `ConnectionManager` (`websocket/manager.py`) assigns each session an `asyncio.Queue`. The WS handler runs a consumer loop (incoming) and producer loop (outgoing) concurrently via `asyncio.gather`.

### Matchmaking via Redis Streams

When a client sends `game_request`:
1. `websocket/endpoint.py` deduplicates via a Redis key `mm:queued:{session_id}` (NX + TTL).
2. It stores request details in a Redis hash (`mm:req:{time_control}:{session_id}`), adds the player to a ZSET (`mm:wait:{time_control}`), and appends to the `mm:requests` stream.
3. The matchmaking daemon (`matchmaking/daemon.py`) reads from `mm:requests` via a consumer group (`mm-requests-daemons`), pairs players, and writes matches to `mm:matches` stream.
4. The API process (TODO: not yet fully wired) will read from `mm:matches` (consumer group `mm-matches-apis`) and send `game_begin` messages to both players via `ConnectionManager.send_to`.

### Frontend Routing

Uses **TanStack Router** with file-based routing. Route files live in `frontend/src/routes/`. The route tree is auto-generated to `src/routeTree.gen.ts` — do not edit it manually.

### API Client Generation

The OpenAPI client in `frontend/src/client/` is auto-generated by `@hey-api/openapi-ts`. Regenerate it after changing backend API shapes with `npm run generate-client`. The `/__schema/ws-messages` endpoint exists solely to expose WebSocket message types to the generator.
